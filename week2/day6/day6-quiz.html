문제 1
객관식 기초
마이크로서비스 아키텍처에서 ConfigMap을 사용하는 주된 이유는?
A. 애플리케이션 코드와 설정을 분리하여 환경별 배포를 용이하게 하기 위해
B. 데이터베이스 비밀번호와 같은 민감한 정보를 안전하게 저장하기 위해
C. Pod의 CPU와 메모리 사용량을 제한하기 위해
D. Pod 간의 네트워크 통신을 제어하기 위해
ConfigMap은 설정 데이터를 저장하는 리소스로, 애플리케이션 코드와 설정을 분리하여 유연한 배포를 가능하게 합니다. 민감한 정보는 Secret을 사용합니다.
문제 2
시나리오 중급
결제 서비스에 Guaranteed QoS를 적용하려고 합니다. 다음 중 올바른 설정은?
결제 서비스는 중요한 서비스로 항상 안정적인 리소스를 보장받아야 합니다.
A. requests만 설정하고 limits는 설정하지 않음
B. requests < limits로 설정
C. requests = limits로 설정
D. requests와 limits 모두 설정하지 않음
Guaranteed QoS는 requests와 limits가 동일해야 하며, 모든 컨테이너에 CPU와 메모리 모두 설정되어야 합니다.
문제 3
주관식 기초
'ecommerce' 네임스페이스에 'db-credentials'라는 이름의 Secret을 생성하는 명령어는?
kubectl create secret generic 명령어를 사용하고, -n 옵션으로 네임스페이스를 지정합니다.
문제 4
코드 분석 중급
다음 Pod 설정에서 사용된 보안 강화 기법은 무엇입니까?
spec: securityContext: runAsNonRoot: true runAsUser: 1000 readOnlyRootFilesystem: true allowPrivilegeEscalation: false
A. 네트워크 정책 적용
B. Pod 보안 컨텍스트 설정
C. 리소스 제한 설정
D. Service Account 권한 제한
securityContext는 Pod나 컨테이너의 보안 설정을 정의합니다. runAsNonRoot, readOnlyRootFilesystem 등은 모두 보안 컨텍스트 설정입니다.
문제 5
주관식 중급
결제 서비스를 전용 노드에만 배치하기 위해 노드에 taint를 추가하는 명령어는?
kubectl taint 명령어를 사용하여 key=value:effect 형식으로 taint를 추가합니다. 예: payment=true:NoSchedule
문제 6
시나리오 고급
API Gateway Pod가 항상 백엔드 서비스 Pod와 다른 노드에 배치되도록 하려면?
고가용성을 위해 API Gateway와 백엔드 서비스가 동일한 노드 장애 시 함께 영향받지 않도록 해야 합니다.
A. nodeSelector를 사용하여 특정 노드 지정
B. podAntiAffinity를 사용하여 분산 배치
C. tolerations를 사용하여 taint 허용
D. resources limits를 다르게 설정
podAntiAffinity는 특정 Pod들이 같은 토폴로지(노드, 존 등)에 배치되지 않도록 하는 기능입니다.
문제 7
코드 빈칸 중급
다음 ServiceAccount 권한 설정에서 빈칸을 채우세요:
apiVersion: rbac.authorization.k8s.io/v1 kind: RoleBinding metadata: name: payment-binding subjects: - kind: ___________ name: payment-sa roleRef: kind: Role name: payment-role apiGroup: rbac.authorization.k8s.io
RoleBinding의 subjects에서 ServiceAccount를 참조할 때 사용하는 kind 값입니다.
문제 8
객관식 중급
readinessProbe와 livenessProbe의 차이점은?
A. readinessProbe는 Pod 재시작, livenessProbe는 트래픽 차단
B. readinessProbe는 트래픽 차단, livenessProbe는 Pod 재시작
C. 둘 다 Pod를 재시작시킴
D. 둘 다 트래픽만 차단함
readinessProbe는 Pod가 트래픽을 받을 준비가 되었는지 확인하고, livenessProbe는 Pod가 살아있는지 확인합니다.
문제 9
주관식 고급
현재 네임스페이스의 모든 Pod의 QoS 클래스를 확인하는 명령어는?
-o custom-columns 옵션을 사용하여 .status.qosClass 필드를 출력합니다.
문제 10
시나리오 고급
트래픽 증가에 따라 자동으로 Pod 수를 늘리려면 어떤 리소스를 사용해야 할까요?
쇼핑몰의 특정 시간대에 트래픽이 급증하여 자동으로 Pod 수를 조절하고 싶습니다.
A. ReplicaSet
B. HorizontalPodAutoscaler (HPA)
C. VerticalPodAutoscaler (VPA)
D. PodDisruptionBudget
HPA는 CPU, 메모리 사용량 등의 메트릭을 기반으로 Pod 수를 자동으로 조절합니다.
문제 11
코드 분석 고급
다음 NetworkPolicy가 수행하는 기능은?
spec: podSelector: matchLabels: app: payment-service policyTypes: - Ingress ingress: - from: - podSelector: matchLabels: app: api-gateway
A. 모든 Pod에서 payment-service로의 접근 허용
B. api-gateway Pod에서만 payment-service로의 접근 허용
C. payment-service에서 api-gateway로의 접근 허용
D. 모든 네트워크 트래픽 차단
이 NetworkPolicy는 payment-service Pod에 적용되며, api-gateway 레이블을 가진 Pod에서만 들어오는 트래픽을 허용합니다.
문제 12
주관식 중급
Rolling Update 중 최대 1개 Pod까지 추가로 생성할 수 있도록 설정하는 필드는?
strategy.rollingUpdate에서 추가로 생성할 수 있는 Pod 수를 제한하는 필드입니다.
문제 13
시나리오 고급
InitContainer를 사용하는 주된 목적은?
Frontend Pod에서 시작하기 전에 정적 파일을 생성하는 작업이 필요합니다.
A. 메인 컨테이너와 함께 동시에 실행되는 Sidecar 구성
B. 메인 컨테이너 시작 전 초기화 작업 수행
C. 네트워크 정책 설정
D. 리소스 사용량 모니터링
InitContainer는 메인 컨테이너가 시작되기 전에 실행되어 초기화 작업을 수행하고 종료됩니다.
문제 14
주관식 고급
특정 Deployment의 Rolling Update 상태를 확인하는 명령어는?
kubectl rollout 명령어를 사용하여 배포 상태를 확인할 수 있습니다.
문제 15
통합 시나리오 최고급
E-Commerce 마이크로서비스 전체 배포에서 가장 중요한 보안 고려사항은?
결제 정보, 사용자 개인정보 등 민감한 데이터를 다루는 마이크로서비스 환경에서 종합적인 보안 전략이 필요합니다.
A. 높은 CPU/메모리 리소스 할당
B. Secret 관리, RBAC, NetworkPolicy, SecurityContext 종합 적용
C. 많은 수의 Replica 설정
D. 빠른 Rolling Update 설정
마이크로서비스 보안은 다층 방어(Defense in Depth) 접근법이 필요합니다. 데이터 보호, 접근 제어, 네트워크 격리, 컨테이너 보안 등을 종합적으로 고려해야 합니다.
