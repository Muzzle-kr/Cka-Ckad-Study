문제 1 객관식
Resource Requests와 Limits에 대한 설명 중 올바른 것은?
A. Requests는 Pod가 사용할 수 있는 최대 리소스를 의미한다
B. Limits는 Pod가 보장받고 싶은 최소 리소스를 의미한다
C. Requests는 스케줄링 시 고려되는 최소 보장 리소스이다
D. Requests와 Limits는 모두 선택사항이며 설정하지 않아도 된다
Requests는 Pod가 보장받고 싶은 최소 리소스이고, Limits는 Pod가 사용할 수 있는 최대 리소스입니다. Requests는 스케줄러가 Pod를 어느 노드에 배치할지 결정할 때 중요한 기준이 됩니다.
문제 2 주관식
CPU 리소스에서 500 밀리코어(millicores)를 코어 단위로 표현하면?
1 코어 = 1000m(밀리코어)입니다. 따라서 500m = 500/1000 = 0.5 코어입니다.
문제 3 객관식
Guaranteed QoS 클래스의 조건은?
A. Requests와 Limits가 모두 설정되지 않은 경우
B. Requests < Limits로 설정된 경우
C. 모든 컨테이너에서 Requests = Limits인 경우
D. Limits만 설정되고 Requests는 설정되지 않은 경우
Guaranteed QoS는 가장 높은 우선순위를 가지며, 모든 컨테이너에서 CPU와 메모리의 requests와 limits가 동일하게 설정되어야 합니다.
문제 4 주관식
노드에 disktype=ssd 레이블을 추가하는 kubectl 명령어는?
kubectl label 명령어를 사용하여 리소스에 레이블을 추가할 수 있습니다. 형식: kubectl label nodes [노드명] key=value
문제 5 코드 빈칸
다음 YAML에서 Guaranteed QoS를 만들기 위해 빈칸을 채우세요:
resources: requests: memory: "200Mi" cpu: "500m" limits: memory: "__________" cpu: "__________"
Guaranteed QoS를 위해서는 requests와 limits가 동일해야 합니다. memory: "200Mi", cpu: "500m"
문제 6 객관식
Node Affinity에서 "requiredDuringSchedulingIgnoredDuringExecution"의 의미는?
A. 스케줄링 시 선택사항이며, 실행 중 변경 시 영향 받음
B. 스케줄링 시 필수조건이며, 실행 중 변경 시 영향 받지 않음
C. 스케줄링과 실행 중 모두 필수조건
D. 스케줄링과 실행 중 모두 선택사항
"required"는 필수조건, "IgnoredDuringExecution"은 실행 중에는 무시된다는 의미입니다. 즉, 스케줄링할 때만 적용되고 실행 중 노드 레이블이 변경되어도 Pod는 유지됩니다.
문제 7 주관식
노드에 "special=true:NoSchedule" taint를 추가하는 명령어는?
kubectl taint 명령어를 사용합니다. 형식: kubectl taint nodes [노드명] key=value:effect
문제 8 객관식
Taint effect 중 "NoExecute"의 특징은?
A. 새로운 Pod만 스케줄링되지 않음
B. 기존 Pod는 유지되고 새 Pod는 스케줄링 가능
C. 기존 Pod도 제거되고 새 Pod도 스케줄링되지 않음
D. Pod 우선순위만 낮춤
NoExecute는 가장 강력한 taint effect로, toleration이 없는 기존 Pod들을 evict(제거)하고 새로운 Pod의 스케줄링도 차단합니다.
문제 9 코드 빈칸
다음 toleration 설정에서 빈칸을 채우세요:
tolerations: - key: "special" operator: "______" value: "true" effect: "NoSchedule"
toleration operator에는 "Equal"과 "Exists"가 있습니다. value가 있으면 "Equal", value가 없으면 "Exists"를 사용합니다.
문제 10 주관식
Pod의 QoS 클래스를 확인하는 kubectl 명령어는?
kubectl get pods에 -o custom-columns 옵션을 사용하여 QoS 정보를 추출할 수 있습니다. QoS 경로: .status.qosClass
문제 11 객관식
Pod Anti-Affinity의 목적은?
A. 특정 노드에만 Pod를 배치하기 위해
B. 특정 Pod와 같은 노드에 배치하기 위해
C. 특정 Pod와 다른 노드에 배치하기 위해
D. Pod의 리소스 사용량을 제한하기 위해
Anti-Affinity는 "반-친화성"으로, 특정 Pod들을 서로 다른 노드에 분산 배치하여 고가용성을 확보하는 데 사용됩니다.
문제 12 주관식
클러스터의 모든 노드의 리소스 사용량을 확인하는 명령어는?
kubectl top 명령어를 사용하여 리소스 사용량을 확인할 수 있습니다. nodes는 노드의 리소스 사용량을 보여줍니다.
